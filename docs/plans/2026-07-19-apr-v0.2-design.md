# Android Project Runtime v0.2 设计

- 日期：2026-07-19
- 状态：已批准
- 原始需求：`APR-Design-v0.2.md`

## 1. 产品目标

APR 是运行在 macOS 上的 Android 确定性验证工具。开发者或 AI Agent 完成代码修改后，APR 负责自动完成真实的构建、安装、启动、用户路径回放、断言、截图、日志采集和报告生成。

APR 的执行与判定过程不使用 AI，不调用模型，也不依赖视觉推理。录制一次后，Journey 可以被开发者、Claude Code 或其他 Agent CLI 低成本、可重复地调用。

APR Journey 是完全自研的 JSON 协议、Recorder、Replay 和断言引擎。它不复用、不调用、也不兼容 Android CLI 官方 Journey。官方 Journey 只作为产品定位对比：官方 Journey 面向 AI 驱动的探索式验证，APR Journey 面向确定性高频回归。

## 2. v0.2 范围

包含：

- macOS
- Android Application Gradle 项目
- 单个 Emulator 或 USB Device
- Android CLI 与 ADB 集成
- 显式 Gradle Build
- APR 交互式 Journey Recorder
- 确定性 Journey Replay
- `click`、`longClick`、`inputText`、`swipe`、`back`、`wait`
- `resourceId`、`text`、`contentDescription` Locator
- Activity 前后状态检查点
- Activity、Element、Logcat 显式断言
- 基于 Layout Diff 的稳定等待
- Screenshot、分步骤 Logcat 和结构化报告
- 面向人类及 AI Agent 的稳定 CLI 契约

不包含：

- APR 内部 AI 推理
- 官方 Journey 的调用或兼容
- OCR、视觉探索、DFS
- Android Studio Plugin
- Windows、Linux、CI
- 多设备并发或常驻 Daemon
- 自动修复 Journey
- 自动推断业务断言
- Claude Code Skill 或 SubAgent 集成
- 监听用户在设备上的任意原始触控
- Accessibility 伴生 App

## 3. 技术与架构

APR 使用 Node.js、TypeScript 和 ESM，采用“核心领域 + Ports/Adapters”架构：

```text
CLI -> Application/Runtime -> Domain <- Ports <- Adapters
```

### 3.1 模块职责

- `domain/journey`：APR Journey 协议、Action、Locator、Expect、Activity Checkpoint、校验规则和结果模型。
- `application/runtime`：编排 Build、Run、Replay、Collect、Report，负责失败中止和结果聚合。
- `application/recorder`：驱动交互式录制，读取 Layout、选择目标、执行操作并生成 Journey。
- `ports`：Gradle、Android CLI、ADB、时钟、文件系统、进程执行等抽象接口。
- `adapters`：基于 Node 子进程实现真实 Gradle、Android CLI 和 ADB 调用。
- `interaction`：解析 Locator，将用户动作转换为 ADB 操作，并管理标注截图兜底。
- `collector`：持续采集 Logcat、步骤切片和截图。
- `report`：生成 JSON 与文本报告并管理运行产物。
- `cli`：参数解析、交互提示、进度输出和退出码映射，不承载业务逻辑。

Domain 与 Runtime 不直接调用 `child_process`。所有外部能力通过 Port 注入，以便使用 Fake Adapter 进行无设备确定性测试。

v0.2 为单进程、单设备、单 Journey 顺序执行。

## 4. CLI 契约与 Agent 使用路径

APR 提供三个主要入口：

- `apr doctor`：检查 Node、ADB、Android CLI、Gradle 项目、权限和设备状态。
- `apr record`：交互式执行设备操作并生成 APR Journey。
- `apr verify`：执行完整验证闭环，是未来 Skill 或 SubAgent 的稳定调用入口。

Journey 准备流程：

```text
apr record
-> APR 交互式执行操作
-> 生成 Journey JSON
-> 开发者或外部 AI 显式补充业务 expect
-> Journey 校验
-> Journey 进入项目版本库
```

代码修改后的验证流程：

```text
Agent 完成开发
-> apr verify
-> Gradle Build
-> Android CLI 安装并启动 APK
-> APR Replay
-> 确定性检查与断言
-> Screenshot + 分段 Logcat
-> 机器可读报告和退出码
```

`apr verify --json` 的 stdout 只输出最终 JSON；进度和诊断写入 stderr。这样外部 Agent 不需要解析人类日志。

稳定退出码：

- `0`：验证通过
- `1`：项目验证失败
- `2`：配置或 Journey 输入错误
- `3`：工具、权限或设备环境错误
- `4`：APR 内部错误

本期不实现 Claude Code Skill，也不由 APR 启动 SubAgent。未来集成仅消费稳定的 CLI 与报告契约。

## 5. 项目配置

项目根目录使用 `apr.config.json`：

```json
{
  "version": 1,
  "build": {
    "task": ":app:assembleDebug"
  },
  "artifact": {
    "target": "app",
    "variant": "debug"
  },
  "run": {
    "packageName": "com.example.app",
    "activity": ".MainActivity"
  },
  "idle": {
    "pollIntervalMs": 200,
    "stablePolls": 2,
    "timeoutMs": 5000
  },
  "artifactsDir": ".apr/runs"
}
```

规则：

- `packageName` 必填，不从文件名或 Activity 猜测。
- `activity` 支持完整类名或以 `.` 开头的相对类名。
- APR 将 Package 与 Activity 规范化为完整 Component。
- Package 用于进程识别、前台 Activity 检查、崩溃识别、相关日志限定和报告元数据。
- 配置与 Android CLI 项目描述或 APK 信息冲突时直接失败，不静默覆盖。
- CLI 参数可以临时覆盖设备、Journey、Package、Activity 和报告路径，但不修改配置文件。

## 6. APR Journey

Journey 描述用户操作以及确定性的录制状态和显式业务预期。

示例步骤：

```json
{
  "action": "click",
  "locator": {
    "resourceId": "toolbar_search"
  },
  "activity": {
    "before": "com.example.app.MainActivity",
    "after": "com.example.app.SearchActivity"
  },
  "expect": {
    "type": "element",
    "locator": {
      "resourceId": "search_input"
    },
    "timeoutMs": 3000
  }
}
```

### 6.1 Activity Checkpoint

Recorder 对每个步骤自动记录：

- 执行操作前的前台 Activity：`activity.before`
- 操作完成且 Layout 稳定后的前台 Activity：`activity.after`

Replay 在 Locator 解析前检查 `before`，在 Action 与稳定等待后检查 `after`。Activity 统一结合配置中的 `packageName` 规范化为完整类名。

Activity Checkpoint 是设备直接观测到的确定性基线，不属于业务语义推断。人工或外部 AI 补充的 `expect` 与它分开处理。

### 6.2 Locator

Locator 字段为：

1. `resourceId`
2. `text`
3. `contentDescription`

Recorder 按上述优先级选择非空且唯一的 Locator。Replay 对零命中和多命中分别报告 `LOCATOR_NOT_FOUND` 与 `LOCATOR_AMBIGUOUS`，不猜测目标。

当常规 Locator 无法定位时，`click` 与 `longClick` 可以使用 `android screen capture --annotate` 和 `android screen resolve` 进行显式降级。由于 `screen resolve` 只能解析标注截图中的 `#编号`，Journey 必须显式保存录制阶段由用户选择的标签：

```json
{
  "fallback": {
    "type": "annotatedLabel",
    "label": "#7"
  }
}
```

Replay 不使用视觉或 AI 猜测标签。没有显式 `fallback` 时，常规 Locator 失败即失败；有显式标签时才采集新的标注截图并解析。报告必须标记降级路径、标签并保存标注截图。其他 Action 不接受坐标兜底。

### 6.3 Action

- `click`：ADB tap
- `longClick`：ADB swipe，起点终点相同并带持续时间
- `inputText`：ADB input text，依赖已聚焦输入目标
- `swipe`：ADB swipe
- `back`：ADB keyevent
- `wait`：仅执行 Layout 稳定检测，不允许固定 sleep

### 6.4 Expect

显式业务断言类型：

- `activity`：前台 Activity 匹配
- `element`：Layout 中出现指定元素
- `logcat`：步骤时间窗口内匹配指定 tag、level 和 pattern

Recorder 不推断或自动生成业务 `expect`。

## 7. 交互式 Recorder

v0.2 使用 APR 交互式录制，不监听任意设备触控：

1. 检查环境并启动目标 App。
2. 读取当前 Layout 并展示可交互元素。
3. 用户选择 Action。
4. 对需要目标的 Action，从 Layout 元素列表选择目标。
5. 对 `click` 或 `longClick`，用户可以查看标注截图并显式选择一个 `#编号` 作为可选降级标签。
6. 读取并记录 `activity.before`。
7. APR 立即执行操作并等待 Layout 稳定。
8. 读取并记录 `activity.after`。
9. 只把实际执行成功的步骤加入内存 Journey。
10. 用户结束后一次性原子写入 Journey JSON，避免半成品。

## 8. Verify Runtime

1. 加载并校验配置与 Journey。
2. 运行 Gradle Wrapper 指定任务。
3. 调用 `android describe` 获取目标与输出产物元数据。
4. 按配置中的 target 与 variant 唯一定位 APK；结果不唯一时失败，不猜测。
5. 启动持续 Logcat 采集。
6. 使用 `android run` 安装并启动已构建 APK。
7. 确认 Package 进程、前台 Activity 和初始 Layout。
8. 建立 Layout Diff 基线。
9. 顺序执行 Journey 步骤。
10. 无论成功或失败，最佳努力采集最终截图、关闭 Logcat 并生成报告。
11. 在临时目录生成产物，最后原子移动到正式运行目录。

### 8.1 单步执行

每一步：

1. 记录单调时钟 `T0`。
2. 检查 `activity.before`。
3. 解析 Locator。
4. 执行 Action。
5. 轮询 `android layout --diff`。
6. 连续 `stablePolls` 次空 Diff 后判定稳定。
7. 检查 `activity.after`。
8. 执行可选业务 `expect`。
9. 记录 `T1` 与 `[T0, T1]` 的步骤日志切片。

任一步失败立即停止后续步骤，避免在错误页面上继续操作。

### 8.2 Layout 稳定等待

- Action 前建立完整 Layout 基线。
- Action 后按 `pollIntervalMs` 轮询 Diff。
- 连续空 Diff 次数由 `stablePolls` 配置。
- 达到 `timeoutMs` 时返回 `IDLE_TIMEOUT`，保存最后一次 Diff。
- 实现使用可注入 Clock/Scheduler，生产和测试均不依赖硬编码 sleep。

### 8.3 Logcat 时间切片

- Replay 前启动一个持续 Logcat 流。
- 每条日志关联接收时的单调时间。
- 每个步骤以 `T0` 与 `T1` 从流中切片。
- Logcat 断言只匹配本步骤窗口内的 tag、level 和 pattern。
- 完整原始日志和步骤日志都保存到报告目录。

## 9. 失败语义

运行最终状态：

- `passed`：Build、Run、Journey、断言与必要采集全部成功。
- `failed`：被验证项目不符合要求。
- `error`：配置、环境或 APR 内部问题导致验证无法正常完成。

固定失败类别：

```text
CONFIG_INVALID
ENVIRONMENT_MISSING_TOOL
DEVICE_UNAVAILABLE
BUILD_FAILED
APP_LAUNCH_FAILED
APP_CRASHED
LOCATOR_NOT_FOUND
LOCATOR_AMBIGUOUS
ACTION_FAILED
IDLE_TIMEOUT
ACTIVITY_BEFORE_MISMATCH
ACTIVITY_AFTER_MISMATCH
EXPECT_ACTIVITY_FAILED
EXPECT_ELEMENT_FAILED
EXPECT_LOGCAT_FAILED
COLLECTION_FAILED
INTERNAL_ERROR
```

原始失败是报告的主失败。失败后的截图、日志或报告辅助采集如果再次失败，只记录为次要错误，不覆盖主失败。

## 10. 报告与产物

每次运行生成唯一 `runId`：

```text
.apr/runs/<timestamp>-<journey>-<runId>/
├── report.json
├── summary.txt
├── screenshot.png
├── logcat.txt
└── steps/
    ├── 001-logcat.txt
    ├── 001-layout-diff.json
    └── 001-fallback-annotated.png
```

`report.json` 包含：

- 报告 Schema 版本、运行时间和时长
- 项目、Package、Activity、Journey 名称和 Journey 内容哈希
- 设备序列号与工具版本
- Build、Run、Structural、Activity Checkpoint、Explicit Expect、Collection 分层结果
- 每一步的 Locator、Action、Idle、Activity、断言和日志切片结果
- 坐标兜底使用记录
- 一个主失败和若干次要错误

所有产物只保存在本地，不上传外部服务。

## 11. 工程依赖与安全边界

- npm 与锁定的 `package-lock.json`
- `commander`：CLI 参数解析
- `@inquirer/prompts`：交互式 Recorder
- `zod`：配置、Journey 与报告边界校验
- `vitest`：测试与覆盖率
- 外部命令基于 Node `child_process`

外部命令必须支持超时与取消。参数使用参数数组传递，不拼接 Shell 字符串。

## 12. 测试策略

### 12.1 Domain 单元测试

覆盖 Journey Schema、Locator 优先级与歧义、Activity 规范化、断言、失败分类和退出码。

### 12.2 Adapter 契约测试

使用临时假命令验证 Gradle、Android CLI 与 ADB 的参数、输出解析、超时、非零退出码和异常输出。

### 12.3 Runtime 集成测试

使用 Fake Ports 验证完整 Build、Run、Replay、Collect、Report，包括失败立即中止、Activity 前后检查、日志时间切片和降级定位。

### 12.4 CLI 黑盒测试

运行真实 APR 进程，验证 `doctor`、`record`、`verify --json` 的 stdout/stderr 隔离、报告产物与退出码。

### 12.5 真实设备验收测试

提供最小 Android 示例项目，包含两个 Activity、可定位元素和确定性业务日志。在 Android CLI 与设备可用时运行端到端 Journey；普通测试不依赖设备。

质量门槛：

- `npm test`
- `npm run typecheck`
- `npm run lint`
- `npm run build`
- Domain 与 Runtime 关键状态机及错误分支全部测试
- 测试不得依赖执行顺序、真实时间或固定 sleep

## 13. v0.2 完成标准

- 可以通过 APR 交互式 Recorder 生成包含 Action、Locator 与 Activity Checkpoint 的 Journey。
- 可以显式补充并校验 Activity、Element 与 Logcat 业务断言。
- 可以自动执行 Gradle Build 和 Android CLI Run。
- 可以通过 ADB 确定性执行全部首期 Action。
- 可以基于 Layout Diff 稳定等待，不使用固定 sleep。
- 可以按步骤切片并匹配 Logcat。
- 可以输出 Screenshot、原始日志、步骤日志和分层报告。
- 可以通过 `apr verify --json` 被任意外部 AI Agent CLI 稳定调用。
- APR 的执行与判定结果不依赖 AI。
